## 1. 시스템 명세 (1주 정찰의 *진짜* 산출물)

### 1.1 Input
4칸 연구 제안서. 예시: T-1 (Molecular Numbering System).
필드: 연구 배경 / 연구 목표 / 연구 방법 / 기대 산출물.
예시:
**연구 주제 T-1.** **분자 번호매기기 시스템 (Molecular Numbering System)**

_정수론적 일대일 대응을 통한 새로운 분자 식별 체계 개발_

|            |                                                                                                                                                                                                                                                             |
| ---------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **연구 배경**  | 분자는 원자 종류, 결합 구조, 입체화학적 정보를 포함하는 복잡한 객체이다. 현재의 분자 식별 시스템(SMILES, InChI, CAS 번호 등)은 선형 문자열 혹은 임의 번호 부여 방식에 의존하여, 수학적 구조가 결여되어 있다. 괴델 번호매기기(Gödel numbering)는 논리식의 집합과 자연수 사이의 일대일 대응을 소인수분해를 이용해 구성한 것으로, 이 아이디어를 분자 구조에 적용하면 수학적으로 엄밀한 분자 번호 시스템을 구성할 수 있다. |
| **연구 목표**  | 분자 그래프(원자 = 꼭짓점, 결합 = 간선)를 정수론적 구조로 인코딩하는 일대일 함수를 정의하고, 이를 기반으로 (1) 분자 간 산술적 연산의 화학적 의미를 해석하고, (2) 동형 분자 판별 알고리즘을 정수론적으로 재기술하며, (3) 새 번호 체계와 기존 InChI/SMILES와의 관계를 수학적으로 분석한다.                                                                              |
| **연구 방법**  | 소수를 원자 종류에 할당하고, 결합 구조 및 입체화학 정보를 지수부에 인코딩하는 방식으로 분자 번호 함수를 설계한다. 중국인의 나머지 정리(CRT) 및 유일 소인수분해 정리(UFT)를 핵심 도구로 활용하며, Python으로 소규모 분자 집합에 대한 프로토타입 구현 및 검증을 수행한다.                                                                                             |
| **기대 산출물** | 정수론적 분자 번호 시스템 이론 논문 1편, Python 프로토타입 코드(공개 저장소), 분자 유사도와 수론적 거리 척도의 상관관계 분석 결과                                                                                                                                                                             |

### 1.2 Output
End-to-end:
- Paper draft (LaTeX 또는 markdown)
- Working code (도메인 적절한 언어)
- Evidence trail (가설 → 시도 → 결과 → 채택/기각 추적)

### 1.3 자율성 모드
- 기본: 완전 자율. 목표만 주고 결과만 확인.
- 옵션: 체크포인트 모드. 가설 선택 / 증명 전략 / 코드 방향 / claim
  강도에서 사용자 승인.

### 1.4 처리 범위
Dry-lab 전체. 도메인별 도구는 plugin/extension으로.

## 2. Fork vs Scratch 결정

### 2.1 정찰의 기술적 발견 (요약)
- Tier A 7개 패키지 깔끔 (Day 7 측정: 대부분 75-100% coverage)
- 두뇌부 판정 로직 교체 비용 작음 (검증 1·3 정정)
- Fork와 vendoring(B′)의 인프라 비용 거의 동일 (검증 2)
- CC 분석은 양방향 노이즈 (Day 4 낙관 + Day 5 비관)

### 2.2 기술적으로는 승자 없음
A/B′ 격차 작음. 결정은 비기술 요인으로.

### 2.3 비기술 요인이 결정
- 일반 mechanistic modeling 프레임워크 목표 → MoAI의 SW 가정이
  시작부터 방해
- 시간 압박 없음 + 취미 성격 → "빠른 baseline"의 가치 없음
- 학습이 결과물 가치만큼 중요 → 직접 설계가 우세

### 2.4 결정: Scratch (B)
- MoAI-ADK는 인터페이스/패턴 reference로만 사용
- Tier A 패키지의 코드 vendoring 안 함
- 본인의 research-compiler 추상화에 맞춰 처음부터 설계
